## 1. 올리브영 영수증 7장 OCR (AI 비서 경로)

**테스트 순서:**
1. PWA 하단 챗봇 FAB(말풍선) 탭
2. 입력창 왼쪽 📎 첨부 → 사진 7장 선택 (실제 올리브영 7장, 할인/쿠폰 포함 영수증 권장)
3. 전송 → 분석 대기 (10~20초)

**Pass 조건:**
- 답변 본문: **"영수증 7장 분석했어요. 지출 7건 추가할까요?"** 같은 **한두 문장 요약**
- 카드 표시: **7장** (각 카드 = 하나의 영수증)
- 각 카드 1줄 요약: `올리브영 · 14,500원 · 품목 N건 · 할인 -1,500원` 형태
- 카드 펼치면 (수정 버튼): items 테이블(품목·정가·수량·합계) + 할인/쿠폰 라인 + 합계 검증 (`정가합 16,000 − 할인 1,500 = 예상 14,500 / 결제 14,500 ✓`)

**Fail 증상:**
- 카드가 **24장** 또는 **1장만** 남음
- 답변 본문에 `"총 N장 분석"` / `"중복/빈 N건 제외"` / `"24건 추출"` 같은 내부 문구 노출
- 모든 카드가 동일 vendor·동일 amount 인데 다른 영수증인데 합쳐짐
- 할인/쿠폰이 별도 expense 카드로 분리됨

**봐야 할 화면/로그:**
- Safari 디버그(Mac 연결 시): Console `[assistant] OCR actions dedupe { raw, kept, dropped, kinds }`
- 카드 확인 ✓ 누르기 전엔 DB 저장 안 됨 — 안심하고 확인 가능

## 2. 독립 영수증 스캔 화면 (smart_import 경로)

**테스트 순서:**
1. 매출 또는 지출 탭 → 영수증 스캔 진입 (또는 챗봇 답변 X — 별도 메뉴)
2. **expense(영수증)** 종류 선택 → 사진 1~3장 업로드
3. AI 분석 후 미리보기

**Pass 조건:**
- 영수증 1장 = 미리보기 카드 1개 (이전엔 품목수만큼 N개 카드 였음)
- 각 카드 내부에 **품목 테이블(정가 기준)** + **할인·쿠폰·포인트** 입력 섹션 + **합계 검증 라인**
- 품목/할인 직접 추가/삭제 가능 (`+ 품목 추가` / `+ 할인 추가` 버튼)
- 결제금액과 정가합 차이 100원+ 면 `⚠ 차이 X원` 표시

**Fail 증상:**
- 한 영수증인데 품목수만큼 카드 N개 분리
- 품목 테이블 / 할인 섹션 안 보임 (옛 UI — amount/vendor/category/memo 만)
- "추가하기" 후 매출 탭에서 N건 expense 늘어남 (1건이어야)

**봐야 할 화면/로그:**
- 백엔드 응답 미리보기 카드 내부 sub-section (회색 배경 박스)
- 저장 후 매출/지출 목록 — 1영수증 = 1 record

## 3. 회원권 충전/사용 (챗봇 신규 action)

**충전 테스트 순서:**
1. 챗봇 입력: **"김서연 회원권에 5만원 충전해줘"** (실제 등록된 고객 이름 사용)
2. 응답 카드 확인 → "추가하기 ✓" 탭

**Pass 조건:**
- 카드 표시: `회원권 충전 / 김서연 · +50,000원`
- 추가하기 후 토스트 `💳 김서연님 회원권 +50,000원 (잔액 N원)`
- **고객 허브 즉시 열어서 김서연 잔액 +50,000 반영 확인** (P0-2 invalidate 검증)

**사용 테스트:**
- "김서연 회원권 3만원 차감" → `회원권 사용 / 김서연 · −30,000원`
- 잔액 부족 시 → 400 응답 (토스트: "잔액 부족 (현재 X원, 필요 Y원)")

**Fail 증상:**
- 카드 안 뜸 / 챗봇이 "어떻게 도와드릴까요" 답변
- 추가 후 고객 허브에 잔액 stale (P0-2 invalidate 누락)
- 잔액 음수 가능

## 4. 예약 노쇼/완료 처리 (챗봇 신규 action)

**테스트 순서:**
1. 캘린더에서 "오늘 2시 김서연" 예약 1건 생성 (없으면 만들기)
2. 챗봇 입력: **"오늘 2시 김서연 노쇼야"** → 카드 확인 ✓
3. 또 다른 예약: **"김서연 시술 끝났어"** (또는 booking_id 명시) → 완료 처리

**Pass 조건:**
- 카드: `노쇼 처리 / 김서연 · 오늘 14시 · 노쇼` (또는 `시술 완료`)
- 추가하기 후 토스트 `📌 김서연 (5/17 14:00) → 노쇼 처리`
- **캘린더 즉시 새로고침 → 해당 예약 색상/상태 변경** (P0-2 invalidate 검증)
- 노쇼는 고객 상세에서 `누적 노쇼 +1` 반영

**Fail 증상:**
- 예약 매칭 실패 ("해당 예약을 찾지 못했어요" 404)
- 캘린더에 status 그대로 (stale)
- 노쇼 카운트 안 늘어남

---

## 5. 환불 처리 (챗봇 신규 action)

**테스트 순서:**
1. 매출 목록에서 임의 매출 1건 ID 확인 (예: #42)
2. 챗봇 입력: **"매출 #42 환불 처리해줘"** 또는 **"매출 #42 환불 — 고객 요청"**

**Pass 조건:**
- 카드: `환불 처리 / 매출 #42 · 고객 요청 · 환불 -50,000원`
- 추가하기 후 토스트 `↩️ 매출 #42 환불 처리 (-50,000원)`
- 매출 목록에 **새 record (음수 -50,000)** 생성 (원본 보존)
- 일/주/월 합계 즉시 반영

**Fail 증상:**
- "어느 매출인지 알려주세요" 되묻기 (LLM 이 revenue_id 추출 실패)
- 원본 매출이 사라짐 (음수 record 가 아닌 hard delete)
- 합계 stale

## 6. 서비스 가격 변경 (챗봇 신규 action)

**테스트 순서:**
1. 가격표 진입 → 현재 "젤네일" 가격 확인 (없으면 추가)
2. 챗봇 입력: **"젤네일 가격 7만원으로 변경"**

**Pass 조건:**
- 카드: `가격 변경 / 젤네일 · → 70,000원`
- 추가하기 후 토스트 `💰 젤네일 가격 변경 X원 → 70,000원`
- **가격표 즉시 새로고침 → 70,000원 반영** (P0-2 invalidate 검증)

**Fail 증상:**
- "서비스를 찾지 못했어요" 404 (service name 정확히 일치 안 함)
- 가격표 stale

## 7. 저장 후 화면 즉시 반영 (P0-2 종합 검증)

각 항목 후 **새로고침 없이** 해당 영역 열어서 stale 여부 확인:

| Action | 즉시 반영 확인 화면 |
|---|---|
| `charge_membership` | 고객 허브 > 해당 고객 > 회원권 잔액 |
| `use_membership` | 동일 |
| `mark_booking_no_show` | 캘린더 / 고객 상세 (노쇼 카운트) |
| `mark_booking_completed` | 캘린더 (상태 색상) |
| `refund_revenue` | 매출 목록 + 오늘/주/월 합계 |
| `update_service_price` | 가격표 / 시술 프리셋 |

**Pass:** 6개 모두 stale 없이 즉시 반영
**Fail 1건이라도:** P0-2 invalidate 누락 — 해당 kind 보고

## 8. PWA 캐시/새 버전 반영 (재진입 후)

**테스트 순서:**
1. PWA 강제 종료 → 다시 열기
2. 1시간 후 다시 열기 (백그라운드 SW update 동작 확인)

**Pass 조건:**
- 빌드 번호 그대로 `v144` (또는 더 최신)
- 부팅 시 흰화면 없음 / SyntaxError 안 뜸
- 차후 사용자가 새 푸시 시 자동 swap 동작

**Fail 증상:**
- "Uncaught SyntaxError: Unexpected end of input" (옛 잘린 캐시)
- 흰화면 / 로딩 스피너 무한
- 빌드 번호 옛 버전

**Fail 시 복구:**
- 설정 → Safari → 고급 → 웹사이트 데이터 → 해당 도메인 삭제
- 또는 홈 화면 PWA 길게 → 삭제 → 재설치
