## 4. 🔐 안전장치 (이게 핵심 — 깨지지 않게)

### 4-A. 현재 리스크 수치 (왜 무섭나)

| 항목 | 개수 |
|------|------|
| `app-*.js` 에서 `getElementById(...)` 사용 | **258개** (중복 제거 전) |
| `app-*.js` 에서 `querySelector(...)` 사용 | **42개** |
| `index.html` 의 class 선언 | **218개** |
| 가장 많은 파일 | `app-caption.js` (94개 셀렉터) |

→ HTML 구조만 살짝 바꿔도 이 중 하나가 깨지면 기능 파괴. 그래서 **P0 선행 감사가 필수**.

### 4-B. P0 · 고정 셀렉터 리스트 (원영님 프롬프트에 없던 선행 단계, 내가 추가 제안)

**담당: T1 Architect (Opus)**
**작업:**
1. 모든 `app-*.js` 파일에서 `getElementById`, `querySelector`, `querySelectorAll`, `classList.add/remove/toggle`, `addEventListener` 대상 추출
2. 중복 제거 후 `.ai/tickets/T-200/SELECTOR_FREEZE.md` 에 기록
3. 각 셀렉터를 3개 카테고리로 분류:
   - 🔴 **필수 보존** (JS 가 직접 참조하는 ID/class) — HTML 리프레시 후에도 반드시 존재
   - 🟡 **동작 보존** (이벤트 위임 대상) — 부모/자식 관계 유지 필요
   - 🟢 **자유** (순수 CSS 장식용) — 바꿔도 됨
4. **추가:** 기존 다크모드 플래그 (`body.dark`, `document.body.classList`, `localStorage.theme` 등) 검출 후 새 `[data-theme="dark"]` 로의 **매핑 규칙** 기록
5. **추가:** `index.html` 내 인라인 `onclick="..."` 존재 여부 전수조사 (§3 #6 위반 있는지)
6. 각 Phase 에서 T2 가 참조할 체크리스트 제공

**산출물:** `.ai/tickets/T-200/SELECTOR_FREEZE.md` (예상 300~500줄)

**완료 기준:** 원영님 🟢 + 오케스트레이터 교차검증

### 4-C. 단계별 안전장치 (모든 Phase 공통)

| 단계 | 누가 | 뭘 |
|------|------|-----|
| 착수 전 | T2 | `wc -l` 로 대상 파일 현재 크기 공유 → 500줄 경계 체크 |
