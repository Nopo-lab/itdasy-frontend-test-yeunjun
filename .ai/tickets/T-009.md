## 3. 위험 시나리오

1. **정상 경로:** `item.id` 는 DB 가 생성한 UUID/해시. sanitize 없이도 문자 제한적이라 안전.
2. **위험 경로:** 만약 백엔드 어느 API 에서 `item.id` 를 클라이언트 입력으로 허용 (예: 유저가 갤러리 업로드 시 파일명을 `id` 로 쓰는 경우) → `"); alert(1); //` 같은 페이로드로 탈출 가능.
3. **2차 공격면:** `item.label`, `item.date` 도 innerHTML 에 sanitize 없이 삽입. DB 에 HTML 태그 포함 라벨이 들어가면 실행됨.

## 4. 수정 방향

### 옵션 A (권장) — `innerHTML` 템플릿 → `createElement` + `addEventListener`
```js
const root = document.createElement('div');
// ...
const btn = document.createElement('button');
btn.textContent = '📸 다시 올리기';
btn.addEventListener('click', () => _republishGalleryItem(item.id));
root.appendChild(btn);
```
- 장점: 구조적으로 XSS 불가
- 단점: 코드 길이 +30~50% · P2 스타일 변경과 충돌 가능성 → **T-200 완료 후 착수**

### 옵션 B — `escapeHtml()` 유틸 + 값만 sanitize
```js
const esc = s => String(s).replace(/[&<>"']/g, c =>
  ({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c]));
pop.innerHTML = `<button onclick="deleteGalleryItem('${esc(item.id)}')">...`;
```
- 장점: 패치 작다
- 단점: 이스케이프 1곳 누락해도 취약 · `onclick="..."` 안에서는 `\'` 가 동작 안 함 (HTML 속성 파싱 선행)

### 옵션 C (최소 리스크) — 백엔드 sanitize 증명만
- T3 가 Supabase RLS/스키마에서 `gallery.id` 생성 규칙 확인
- UUID/해시만 허용된다는 증거 → 본 티켓 **문서화로 종료** (코드 수정 없음)

**추천:** 옵션 C 로 현재 안전성 먼저 증명 → 중장기로 옵션 A 리팩터.
