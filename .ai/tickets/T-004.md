**위험 라벨:** `risk:integration`

## 목표

현재 백엔드가 HTTPException 을 일관되지 않은 포맷으로 반환 → 프론트에서 에러 메시지 표시 분기 복잡. `{ code, message }` 또는 `{ code, message, details }` 포맷으로 통일.

## Acceptance Criteria

- [ ] 모든 HTTPException 응답이 `{ code: string, message: string }` 구조
- [ ] code 는 `AUTH_FAILED`, `QUOTA_EXCEEDED`, `RLS_DENIED`, `VALIDATION_ERROR` 등 enum-like 문자열
- [ ] 프론트가 파싱 실패해도 graceful (예전 포맷 호환)
- [ ] 스테이징 Railway 배포 후 프론트 (스테이징) 에서 주요 API 5개 호출해서 새 포맷 확인

## 영향 API (우선 5개)

- `POST /caption/generate`
- `POST /instagram/publish`
- `POST /portfolio`
- `PUT /shop/persona/identity`
- `GET /portfolio`

## 연동 체크리스트

- [ ] 프론트의 기존 에러 파싱 코드가 신 포맷 호환하는지 T2 와 조율
- [ ] Supabase RLS 에러 메시지를 사용자에게 **직접 노출 금지** (code 로 래핑)
- [ ] Sentry/로깅에는 원본 에러 유지

## 주의

이 티켓은 프론트 레포 작업이 아님. T3 가 다른 레포에서 수행.
완료 후 프론트에서는 에러 메시지 매핑 테이블 T-004b (후속 티켓) 로 분리.
